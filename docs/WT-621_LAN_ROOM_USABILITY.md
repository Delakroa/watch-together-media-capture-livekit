# WT-621 — LAN room usability clarity

## Статус

Завершено.

## Цель

Убрать две ложные причины сомнений при проверке комнаты между домашними
компьютерами: нечитаемые технические события в тёмном workspace и ошибочное
ощущение, что голосовой чат сломан в обычном HTTP LAN-режиме. Дополнительно
предупредить host-а, когда invite с `localhost` физически нельзя открыть с
другого устройства.

## Реализация

- Активный workspace явно задаёт светлый цвет строк `События`; время остаётся
  вторичным, но читаемым.
- При `isSecureContext === false` карточка голоса показывает статус `Нужен
HTTPS`, объясняет реальную границу и не показывает неработающий CTA
  `Повторить` или `Включить микрофон`.
- Приглашение с `localhost`, `127.0.0.1` или `::1` сопровождается спокойным
  предупреждением: такое приглашение работает только на host-компьютере.
  Интерфейс подсказывает открыть приложение по `http://<IPv4-host>:8088` и
  создать новую комнату.
- Проверка loopback вынесена в `share-invite.ts`; она работает только с уже
  canonical public room URL и не добавляет в invite секреты или query-параметры.

## Проверка

```bash
pnpm --filter @watch-together/frontend exec vitest run src/features/rooms/share-invite.test.ts src/pages/HomePage.test.tsx
pnpm check
pnpm test:e2e
```

Тесты подтверждают распознавание loopback invite, LAN-состояние голоса без
ложной retry-кнопки и сохранение обычного room/create/join flow.

## Ограничения и следующий шаг

- Browser не может безопасно узнать private IPv4 host-а, поэтому UI не
  подставляет IP автоматически и не маскирует неверно настроенный Windows
  Firewall, AP isolation или `lan.env` с IP другого компьютера.
- Для межкомпьютерной проверки host должен запускать стек с собственным
  `LIVEKIT_NODE_IP`, `LIVEKIT_URL` и `PUBLIC_LIVEKIT_URL`, открыть сервис по
  своему IPv4 и только затем создать комнату.
- Голос между разными устройствами остаётся задачей TLS-staging из WT-610:
  HTTP LAN намеренно проверяет room, чат и local-media path без доступа к
  микрофону.
